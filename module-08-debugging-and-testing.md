# 모듈 8: 디버깅과 테스팅(Debugging and Testing)

**목표:** AI의 도움을 받아 에러를 찾고 고치며, 안전한 테스트 코드를 작성하는 비법 배우기

**예상 소요 시간:** 45-55분

---

## 무엇을 배우게 되나요?

이 모듈에서는 개발자의 귀중한 머리카락을 가장 많이 지켜줄 두 가지 필수 스킬, **디버깅(에러 잡기)**과 **테스팅(방어막 치기)**에 대해 다룹니다. 에러 메시지를 어떻게 읽어내야 하는지, 논리적이고 체계적인 디버깅 전술이 무엇인지, 그리고 단위(Unit) 테스트와 통합(Integration) 테스트를 아주 쉽게 작성해내는 방법들을 배킵니다. 지금 이걸 체화하면 코딩 구력이 완전히 달라집니다.

## 레슨 1: 무자비한 에러(Error) 메시지 해석하기

### 에러 메시지의 해부학

에러가 터져서 빨간 글씨가 화면을 덮으면 딱 4가지만 찾으세요:

- **무엇이(What)** 터졌나 (에러의 부류)
- **어디서(Where)** 터졌나 (에러가 난 파일명과 라인 번호 줄)
- **왜(Why)** 터졌나 (가끔 운 좋으면 친절하게 적혀있음)
- **스택 트레이스(Stack trace)** -- 코드가 궤멸할 때까지 밟아온 최후의 발자취들 (맨 위부터 아래로 파고들어간 경로)

---

### 에러 메시지 뜯어보기 예시

```
TypeError: Cannot read property 'name' of undefined
    at getUser (/app/services/userService.js:15:24)
    at /app/routes/users.js:42:18
    at Layer.handle [as handle_request] (/app/node_modules/express/lib/router/layer.js:95:5)
```

**[해부 결과]**
**에러 종류:** `TypeError` -- 타입이 안 맞거나 값이 이상하다는 뜻입니다.
**심장 마비 원인:** `Cannot read property 'name' of undefined` -- 뭔가 정의되지 않은 텅 빈(undefined) 녀석한테서 강제로 `.name` 속성을 끄집어내려다가 시스템이 멈췄습니다.

**죽기 직전의 최후 발자취 (스택 트레이스):**

- 라인 1번: `userService.js` 파일의 15번째 줄에서 사망.
- 라인 2번: 얘가 원래 `users.js` 파일 42번째 줄에서 핑퐁으로 호출된 놈임.
- 라인 3번: Express 프레임워크 자체 시스템 통신 코드 (이런 건 무시해도 됨).

---

### 아, 복잡하네? 그럼 Claude Code한테 싹 다 던지세요

이 모든 고민을 할 필요조차 없습니다. 터진 에러 창을 긁어서 Claude에게 가져다 바치세요:

```
나 지금 이 에러 떴다:
TypeError: Cannot read property 'name' of undefined
    at getUser (/app/services/userService.js:15:24)

이게 도대체 무슨 외계어인지 초등학생 관점에서 원인을 설명하고, 내 코드 어딜 어떻게 뜯어야 병이 낫는지 처방전을 내려줘.
```

그러면 Claude Code가 "친절하게 당신의 빈 값(undefined)이 어디서 구멍 났는지"를 설명하고 패치 코드까지 던져줍니다.

## 레슨 2: 뇌동매매 금지, "체계적 디버깅" 전략

### 제발! 막 고치지 마라! 프로세스(Process)

에러가 떴다고 당황해서 애꿎은 코드를 아무렇게나 고치지 마십시오. 아래 과정을 꼼꼼히 밟으세요.

**1. 범인 소환 (에러 재현하기)**

```
유저가 로그인할 때마다 앱이 종종 뻗는데,
이 에러가 어떻게 하면 100% 뜨는지 확실히 에러 재현용 재물(테스트 환경)을 강제로 재현해 줘.
```

**2. 구역 통제 (문제 고립시키기)**

```
로그인 프로세스에서만 터진다는 걸 잡았다.
자, 아까 그 로그인 함수 싹 다 펼치고 한 땀 한 땀 로직 거꾸로 뜯어서 올라가 보자.
```

**3. 사인 규명 (원인 찾기)**

```
이 함수 내용물이 뻗었다면 무슨 변수에서 터진 걸로 보여? 구멍 난 곳 의심되는 거 다 짚어봐.
```

**4. 수술 집도 (고치기)**

```
네 지적대로 유저 객체가 텅 비었을(Null/Undefined) 땐 다음 줄로 못 넘어가게 미리 컷 시켜버리는 방어(Validation) 로직 껴놔.
```

**5. 확인 사살 (검증)**

```
다 고쳤지? 테스트 봇 돌려서 다른 데 부러진 데 없는지 확실히 검사해 봐.
```

---

### 터진 로그 찾기 최강자: Console.log 투하 전술

가장 원시적이지만 아직도 심장수술 레벨로 효과적인 디버깅 기술입니다.

```
이 함수가 도대체 속에 무슨 값을 품고 터졌는지 1도 모르겠어.
login 함수 여기저기에 중간중간 console.log 잔뜩 심어봐:
- 함수 맨 처음 부를 때 들어온 파라미터 값들 출력
- DB 긁어왔 결과물 데이터 화면에 토해내기
- 계산 중 나오는 모든 임시 변수들 상태 좀 보자
```

나중에 해결되면:

```
좋아 고쳤어. 방금 남발한 console.log 쓰레기들 찾아서 싹 다 지워.
```

## 레슨 3: 단위 테스트 (Unit Tests)

### 단위 테스트란?

코드의 제일 꼬맹이 부품들(함수 하나, 파일 하나)이 격리된 채로 '제 역할'을 완벽하게 100% 수행하는지 검사하는 방어막 설계입니다. 이거 안 깔고 건물 올리면 나중에 피눈물 납니다. 이거 다 짜놓고 올리면 다른 미친놈이 와서 코드 수정하다 좆되는 걸 1초 만에 경고해 줍니다.

### Claude Code 로 테스트 코드 1초 만에 찍어내기

```
calculateTotal 이 합계 구하는 함수 테스트(방어막) 코드 Jest 프레임워크로 싹 다 찢어발겨서 짜놔:

- 장바구니에 양수 아이템들 있을 땐 제대로 더하는지 테스트 (정상 작동)
- 텅 빈 장바구니에선 0 뱉는지 테스트 
- 할인 쿠폰 먹였을 때 퍼센트 계산 깎여서 들어가는지 테스트
- 만약 장난으로 음수형 금액 넣으면 바로 에러(Throw) 터지면서 튕겨내는지 모조리 테스트 해.
```

결과물: `calculateTotal.test.js` 파일이 뚝딱 태어나고 안에는 4개의 방어 시나리오 코드가 전부 구현됩니다.

### 터뜨려볼까? 기상천외한 엣지 케이스 추가

착한 사람들만 고려하지 마십시오. 괴상한 변수(Edge Cases)도 다 테스트 코드에 쏟아부어야 합니다.

```
추가로 이 악마 같은 상황들도 calculateTotal 테스트 코드에 심어 놔:
- 조단위 백경 단위 무지막지하게 큰 금액이 들어왔을 때 오버플로우 나는지
- 소수점 10자리수 같은 이상한 값이 왔을 때
- 아예 입력을 null이나 undefined로 빈값을 던졌을 때
- quantity(수량) 란을 아예 빼먹었거나 "0" 으로 입력했을 때
```

## 레슨 4: 통합 테스트 (Integration Tests)

### 통합 테스트란?

단위 테스트가 바퀴랑 핸들을 점검하는 거라면, 이건 부품들을 조립해서 차가 엔진을 타고 굴러가는지 'API 연동 흐름 전체'를 쏘고 테스트하는 검역소입니다. API 호출, 데이터베이스 저장 같은 걸 통째로 시험합니다.

### 닥치고 백엔드 API 종합 검진 시키기

```
/api/users 엔드포인트(API 주소) 전체를 Supertest 라이브러리 써서 아작나게 통합 테스트 코드로 작성해 봐.

테스트 요건:
- GET /api/users 부르면 진짜 유저 목록 쫙 긁어오는지
- POST /api/users 때리면 유저 실제로 1명 DB에 잘 박히는지
- POST 때 이메일 항목 누락시키면 제대로 400 Bad Request 에러 뱉는지
- DELETE 때리면 진짜 삭제되는지
```

클로드가 테스트 전 전용 DB를 초기화(Clear)하고, 가짜 호출 쏴서 200/400 응답까지 검증하는 시나리오를 완성해줍니다.

## 레슨 5: TDD (Test-Driven Development / 테스트 주도 개발법)

### TDD가 먼데?

코드 짜놓고 뒤늦게 "이거 맞나?" 하고 테스트 짜는 후진 짓을 뒤집은 겁니다.
**"본 코드 한 줄 안 짠 상태에서 방어막(테스트)부터 먼저 짜는, 변태적이지만 전 세계 엘리트들이 다 쓰는 방법론"** 입니다.

1. 무조건 실패하는 테스트 먼저 작성
2. 이걸 간신히 통과할 최후의 미니멀 본 코드 작성
3. 코드 살 붙여서 예쁘게 정리(Refactor)

### 클로드랑 TDD 한 판 붙기

**스텝 1: 백지상태에서 테스트부터 설계**

```
비밀번호 보안성 검증하는 함수를 만들어 볼 건데.
일단 지금 함수 본체 코드는 1도 짜지 말고, 아래 룰들을 만족하는 빈껍데기 '테스트 로직 파일' 부터 먼저 무조건 설계해서 던져 줘:
- 8글자 룰 검증
- 대문자, 특수기호 1개 필수 룰 검증 등등
```

**스텝 2: 실패 확인 (빨간불)**

```
테스트 돌려봐
(당연히 본 함수를 안 짰으니까 개터집니다)
```

**스텝 3: 함수 채워 넣고 해결 (파란불)**

```
자 이제 아까 네가 짰던 저 빨간불 테스트를 전부 다 초록불로 패스시킬 수 있게 '진짜 본문 함수' 로직을 작성해서 끼워 놔!!
```

**스텝 4: 예쁘게 정리**

```
통과했냐? 와 개쩐다 근데 방금 네가 짠 함수 코드가 스파게티 같으니까 깔끔하게 튜닝해서 다이어트 좀 시켜놔. 물론 테스트 코드는 그대로 안 깨지게 유지하고!!!
```

## 흔해빠진 에러 대처 전술 (디버깅)

**시나리오 1: 뻑하면 만나는 Null/Undefined 형제**

```
"Cannot read property 'x' of undefined"
"아 x발 진짜 null에서 못 꺼낸대 클로드야 제발!!"
도와줘:
1. 어디가 빵꾸난 변수인지 추적
2. null 들어오면 통과 안 하게 안전망 짜
3. 에러 안 터트리고 우회해서 빠져나가게 해줘
```

**시나리오 2: 로직 버그 (빨간 줄 하나도 없는데 계산값이 좆망임)**
빨리 잡아야 합니다.

```
장바구니 총액이 25달러인데 얘 계속 30달러로 지 혼자 계산해!
calculateTotal 이 새끼 당장 해부실 올려서:
1. 계산 과정마다 숫자 뭐 나오는지 로그 다 찍어놔
2. 가짜 데이터 쑤셔넣고 한줄한줄 시뮬레이션 돌려
3. 어디서 5달러가 혼자 뻥튀기되는지 도둑놈 잡아내
```

**시나리오 3: 성능 이슈 (돌아가긴 하는데 대시보드가 5초나 걸려)**

```
야 대시보드 로딩 5초 걸려 사람 미치라고 짰냐!!
도와줘:
1. 어떤 미친 DB 쿼리가 병목 걸리는지 추적해
2. 어디 N+1 문제 같은 아둔한 짓거리 하는지 잡아내
3. 당장 최적화(인덱싱 등) 돌려서 해결할 제안을 2개 짜 와! 5초 → 0.5초로 만들어!
```

## 레슨 6: 테스트 돌릴 때 가짜 껍데기 씌우기 (Mocking)

진짜 테스트를 돌리는데, 테스트할 때마다 실제 회원가입 환영 이메일이 내 이메일함으로 수만 개 복사되면 미치겠죠? 그래서 가짜 샌드백(Mock)을 때려보는 겁니다.

```
sendEmail 함수 테스트 짤 건데, 이거 돌릴 때 진짜로 네이버/구글 메일 서버로 쏘면 안 된다!!
실제 이메일 전송 로직 부분은 Mock(가짜 모형 개체)으로 덮어 씌워서, 실제론 안 날아가는데 내부적으론 "어 이메일 쐈어요~" 하고 성공 신호 잘 떨어지는지만 테스트하게 격리 시켜줘!
```

## 모듈 8 요점 체크리스트

모듈 9로 다이브(Dive)하기 전 점검:

- [ ] 에러 메시지의 4가지 해부 구조를 파악하고 AI에게 던져 번역기를 돌릴 수 있다.
- [ ] 에러 났다고 코드부터 조지지 않고 재현, 고립, 통제하는 디버깅 마인드를 갖췄다.
- [ ] 내 코드를 보호해줄 함수별 '단위 테스트'를 AI에게 창조시킬 수 있다.
- [ ] API 대검열을 위한 전체 '통합 테스트' 스펙을 AI가 쓰게 던질 수 있다.
- [ ] 테스트를 먼저 설계하는 진짜 고수들의 TDD 기술을 안다.
