# 모듈 4: 파일 및 코드 작업하기

**목표:** Claude Code로 기존 코드를 읽고, 작성하고, 수정하는 효과적인 방법 배우기

**예상 소요 시간:** 40-50분

---

## 무엇을 배우게 되나요?

이 모듈에서는 매일 가장 자주 하게 될 핵심 작업들을 다룹니다. 기존 코드를 해석하고, 새 코드를 작성하고, 아무것도 부수지 않고 안전하게 코드를 수정하며, 리팩토링하고 시스템을 탐색하는 기술을 익힙니다. 이 기술들에 익숙해지면 허둥대지 않고 놀라울 만큼 생산성이 올라갑니다.

---

## 레슨 1: 코드 읽고 이해하기

### 코드를 읽는 기능이 왜 중요한가요?

프로그래밍의 80%는 코드를 작성하는 것이 아니라 '읽는' 시간입니다. 기능이 어떻게 작동하는지 이해하고, 버그의 위치를 추적하며, 기존 코드의 패턴을 배우기 위해 코드를 읽습니다. Claude Code의 도움을 받아 코드를 잘 읽어내는 능력은 결국 다른 모든 작업의 속도를 폭발적으로 높여줍니다.

### 단일 파일 읽기

특정 파일을 읽고 설명해 달라고 요청하세요:

```
server.js 파일을 읽고 이 파일의 역할을 설명해줘
```

특정 구역만 콕 집어 물어볼 수도 있습니다:

```
auth.js 파일의 login 함수를 읽고 로직이 어떻게 흘러가는지 설명해봐
```

구조를 요약해달라고 지시할 수도 있습니다:

```
package.json을 읽고 이렇게 알려줘:
- 사용 중인 디펜던시 목록
- 실행 가능한 script 종류들
- 메인 구동(엔트리) 포인트
```

### 연관된 여러 파일 한 번에 읽기

기능이 여러 파일에 흩어져 있을 때는 파일들을 콕 집어 요청하세요:

```
유저 로그인 인증이 어떻게 작동하는지 파악하고 싶어.
아래 파일들을 전부 읽고:
- routes/auth.js
- middleware/auth.js
- models/user.js
전체적인 인증 흐름(Flow)을 하나로 연결해 설명해 줘
```

### 프로젝트 뼈대 구조 이해하기

전체적인 빅픽쳐를 보고 싶을 때 유용합니다:

```
이 프로젝트 폴더 구조를 파악하게 도와줘.
- 디렉토리(폴더) 구조 트리 레이아웃
- 각 폴더 안엔 어떤 역할의 파일들이 있는지
- 처음에 앱이 켜지는 진입점 파일
- 코드가 어떤 철학으로 정리되어 있는지
```

Explore 에이전트를 시켜 깊게 파고들 수도 있습니다:

```
Explore 에이전트를 띄워서 전체 코드베이스를 싹 스캔하고 다음을 보고해:
1. 전반적인 아키텍처
2. 핵심 컴포넌트(부품)들
3. 사용된 기술 스택
4. 시스템 전반의 데이터 흐름
```

### 기존 패턴과 규칙 읽어내기 (가장 강조하는 부분)

코딩을 시작하기 전에 이 코드베이스의 기존 규칙을 먼저 배우세요:

```
이 프로젝트에서는 '에러 핸들링'을 주로 어떤 패턴으로 처리하고 있는지 보여줘
이 앱의 API 라우트(EndPoints) 만드는 기존 룰과 패턴을 하나 보여줄래?
데이터베이스 쿼리문은 주로 어디서 어떻게 모아서 쓰고 있어?
```

이런 방식을 쓰면 기존 코드를 존중하면서, 튀지 않고 남들이 쓰던 부품을 그대로 우려먹어 깔끔하게 개발할 수 있습니다.

## 레슨 2: 새 코드 작성하기

### 프로젝트 규칙(Conventions) 준수하기

새 코드를 작성하기 전, 반드시 기존 룰을 물어보세요:

```
내가 새로운 기능을 추가하기 전에, 다음 내용부터 한 번 훑어서 알려줘:
- 이 프로젝트의 컨트롤러와 라우터 파일 구조 패턴
- 변수나 파일명 짓는 Naming 컨벤션 룰
```

그다음에 이렇게 시키세요:

```
/api/users에 있던 기존의 작성 규칙 패턴을 완벽히 똑같이 흉내내서 /api/products 새 라우트 코드를 작성해.
```

### 깔끔하고 구체적인 코드 작성 지시

요청은 구체적일수록 결과가 좋습니다:

```
이메일 주소가 올바른 형식인지 검증하는 함수를 만들어줘:
- 변수 이름은 누가 봐도 직관적으로
- 입력값이 이상하면 튕겨내는 방어(Validation) 로직 추가
- 에러가 나면 뱉을 에러 메시지
- 함수 위에 JSDoc 주석 설명 달기
- 나중에 단위 테스트하기 편한 구조로
```

### 모듈러(조립식) 코드 작성 지시

처음부터 역할을 분리해 달라고 지시하세요:

```
다음을 처리하는 'user service' 모듈을 만들어:
- 유저 생성, 조회, 수정, 삭제 기능
- 데이터베이스 건드리는 로직이랑 비즈니스 로직을 분리해줘
- 작성한 함수는 다 바깥으로 Export 시켜
```

### 주석/문서화(Documentation) 지시

나중에 주석 달려면 귀찮습니다. 처음 짤 때 주석도 같이 달라고 지시하세요:

```
calculateTax라는 함수를 만들건데 아래 주석도 꼼꼼히 적어 놔:
- JSDoc 스타일의 주석
- 파라미터 값들에 대한 설명
- 최종 반환(Return) 값에 대한 설명
- 어떻게 쓰는지 주석 내부에 사용 예시 코드
```

## 레슨 3: 기존 코드 수정하기 (Editing)

### 정밀한 핀포인트 수정

Claude Code의 Edit 도구는 파일 전체를 덮어쓰지 않고 타겟만 정확히 바꿉니다:

```
config.js 파일에서 데이터베이스 포트 번호를 5432에서 3306으로 수정해
```

결과는 이렇게 차이점(diff)만 보여줍니다:

```diff
Old:
const DB_PORT = 5432;

New:
const DB_PORT = 3306;
```

기존 코드에 로직 추가하기:

```
login 함수 내부 코드 중에, 패스워드가 최소 8자리 이상인지 체크하는 방어 로직을 끼워 넣어
```

---

### 안전하게 수정하기

큰 작업을 하기 전에는 Claude Code에게 시뮬레이션(Plan)을 먼저 시켜보세요:

```
데이터베이스 연결부에 에러 핸들링 코드를 추가하고 싶은데, 진짜로 파일을 수정하진 말고 니가 어디어디를 어떻게 바꿀건지 계획만 먼저 나한테 보여줘봐.
```

눈으로 확인 후 승인하세요:

```
계획 좋아 보이네, 그대로 실제 파일들에 적용해서 수정해.
```

### 여러 파일을 한 번에 묶어서 수정하기

```
User 모델 부분에 TypeScript 타입(Type) 선언을 추가할 거야.
다음 파일들 싹 다 찾아서 한 번에 업데이트 해:
- models/user.js (이건 user.ts로 이름 바꿔서)
- 기존에 User.js를 import 해서 쓰던 모든 파일들 경로 수정
- 관련된 Type definition 선언 파일들
```

## 레슨 4: 코드 리팩토링 (Refactoring)

### 중복 코드 제거하기

```
라우트 파일들을 싹 뒤져보고 내용이 완전히 겹치거나 중복되는 코드가 있으면 찾아내
```

이렇게 찾은 후:

```
routes/users.js랑 routes/posts.js 에 똑같이 들어있는 그 검증 로직을 쭉 뽑아내서, 둘 다 공용으로 가져다 쓸 수 있게 재사용 가능한 모듈 파일 하나로 빼내 줘.
```

### 크고 뚱뚱한 함수 쪼개기

```
processOrder 함수가 무려 150줄이나 돼서 읽기가 너무 힘들어.
이걸 의미 단위로 잘게 쪼개서 분리된 작은 함수들로 파편화시켜 줘. 동작은 100% 예전이랑 똑같아야 해.
```

### 최신 문법으로 업데이트

옛날 남의 코드를 만진다면:

```
auth.js 파일을 최신 문법으로 리팩토링 해:
- 콜백(callback) 떡칠된 거 다 async/await 로 세련되게 바꿔
- var 쓴 거 다 const 나 let 으로 바꿔
- 가능하면 화살표 함수 방식으로 바꿔
```

## 레슨 5: 거대한 코드베이스 탐색하기

### 코드 탐색 검색 (Grep/Glob)

텍스트/코드 내부 조각 찾기(Grep):

```
sendEmail 이라는 함수가 도대체 어디서 호출되어 쓰이고 있는지 싹 다 잡아내
코드 어딘가에 박혀있는 TODO 주석들 내용만 다 검색해서 리스트 뽑아 줘
```

파일 껍데기 이름으로 찾기(Glob):

```
프로젝트 안에 있는 테스트 파일들 다 찾아
src 디렉토리 안에 있는 TypeScript 파일만 골라서 찾아
환경 설정용 config 파일들 다 찾아내
```

### 코드 실행 흐름(Flow) 추적하기

```
사용자가 로그인 버튼을 눌렀을 때의 런타임 코드 흐름을 쫙 추적해줘.
API 엔드포인트부터 시작해서 데이터베이스 쿼리가 날아가는 마지막 끝자락까지 스텝별로 묘사해 봐.
```

### 종속성(Dependencies) 및 족보 파악

이걸 바꿔도 안전한지 파악할 때 필수입니다:

```
User 계정 모델(Model)을 Import 해서 영향받는 다른 파일들이 총 몇 개나 돼?
authenticateUser 이 함수를 갖다 쓰는 녀석들 싹 다 추적해 (LSP 도구 사용)
```
