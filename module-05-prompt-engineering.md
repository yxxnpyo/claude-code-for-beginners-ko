# 모듈 5: 코딩 작업을 위한 효과적인 프롬프팅

**목표:** 코딩 작업을 AI에게 효과적으로 전달하는 기술 마스터하기

**예상 소요 시간:** 30-40분

---

## 무엇을 배우게 되나요?

이 모듈은 Claude Code를 다루는 데 있어 가장 핵심적인 스킬인 "좋은 프롬프트 작성법"에 대해 다룹니다. AI가 던져주는 코드의 품질은 당신이 입력하는 프롬프트의 퀄리티와 정확히 일치합니다. 구체적으로 말하는 법, 코드에 문맥(Context)을 불어넣는 법, 거대한 작업을 작게 쪼개는 법 등을 배웁니다.

---

## 레슨 1: 좋은 프롬프트의 구조

### 무엇이 좋은 코딩 프롬프트인가요?

좋은 프롬프트는 '구체적(Specific)'이며, '문맥(Contextual)'이 포함되어 있고, '실행 가능(Actionable)'하며, 결과물을 '검증(Testable)'할 수 있어야 합니다.

실제 예시로 비교해 볼까요:

#### 예제 1: 단순한 함수 만들기

**최악 (Bad):**

```
함수 하나 만들어
```

**보통 (Better):**

```
두 숫자를 더하는 자바스크립트 함수 만들어줘
```

**최고 (Best):**

```
addNumbers라는 자바스크립트 함수를 만들어줘:
- a와 b, 두 개의 파라미터를 받음
- 두 숫자의 합을 Return 할 것
- 두 입력값이 모두 숫자인지 미리 검증(Validation)하는 방어 로직 포함
- 친절한 JSDoc 주석 포함
```

#### 예제 2: 버그 고치기

**최악 (Bad):**

```
버그 좀 고쳐줘
```

**보통 (Better):**

```
로그인 버그 고쳐봐
```

**최고 (Best):**

```
auth.js에 있는 login 함수가 빈 패스워드를 제대로 걸러내지 못하고 있어.
유저가 패스워드 칸을 비우고 제출하면 앱이 "Cannot read property 'length' of undefined" 에러를 뿜으면서 서버가 뻗어버려. 
빈 패스워드일 경우 안전하게 잘라내고, 사용자에게 적절한 에러 안내 문구를 반환하도록 방어 로직 코드를 추가해 줘.
```

마지막 지시는 밤과 낮의 차이입니다. 증상, 기대하는 결과, 터진 위치 등을 자세히 줄수록 Claude Code는 1초 만에 백발백중의 정답을 가져옵니다.

## 레슨 2: 기술적 요구사항을 매우 구체화하기

### 기술 및 언어 스택 명시하기

**애매함:** `웹 서버 하나 구축해 줘`

**구체적:**

```
TypeScript와 Express.js를 써서 웹 서버 하나 만들어 줘:
- CORS 허용 셋업
- JSON body 파싱 허용
- Morgan 패키지 써서 요청 로깅
- 통합 에러 핸들링 미들웨어 포함
- 포트 3000 사용
```

### 구조 및 아키텍처 명시하기

**애매함:** `데이터베이스 하나 붙여줘`

**구체적:**

```
SQLite 데이터베이스를 붙여줘:
- better-sqlite3 라이브러리 사용
- DB 파일 생성 경로는 ./data/app.db
- users 테이블(id, username, email, created_at 속성) 생성
- /db/connection.js에 DB 연결 헬퍼 파일 분리
- DB 마이그레이션 스크립트 파일도 세트에 포함할 것
```

## 레슨 3: 문맥(Context) 불어넣기

### 프로젝트 구조적 문맥

**수정 지시 시 문맥 제공:**

```
이 프로젝트는 라우트 기능들이 /routes 폴더 안에 모여있는 Express.js 기반의 API 서버야.
기존 /routes/posts.js 파일에 썼던 패턴 스타일 양식을 그대로 베껴서, 이번엔 데이터베이스에서 모든 사용자를 긁어다 주는 /api/users 새 라우트를 하나 추가해 줘.
```

"기존 파일 패턴 스타일을 흉내 내라"라는 말 한마디가 AI로 하여금 당신의 기존 프로젝트 컨벤션에 100% 녹아들게 만드는 핵심 마법입니다.

### 비즈니스적 문맥 (비하인드 스토리)

AI에게 도메인 지식을 설정하면 스스로 똑똑한 예외 처리를 챙깁니다:

```
새로운 사용자 가입 폼에 검증(Validation) 기능을 달아:
- 이메일은 중복되지 않고 유효한 형식일 것
- 비밀번호 검증 룰: 8자리 이상 + 숫자 1개 + 특수문자 1개 필수
- 18세 이상만 가입 가능 (우리 서비스는 금융 결제 앱 시스템임)
```

마지막에 무심코 던진 "우리는 금융 결제 앱이야"라는 도메인 정보 때문에 Claude는 평소보다 훨씬 깐깐하고 빡센 보안, 컴플라이언스 기준 예외 로직을 스스로 고민해서 제안할 확률이 높아집니다.

## 레슨 4: 이터러블(반복/점진적) 단기 개발법

엄청나게 거대한 기능을 프롬프트 단 한 줄로 해결하려 들지 마세요. 터지면 원인도 모릅니다.

**1단계: 가장 멍청한 베이직 버전**

```
터미널에서 도는 초간단 할 일(Todo) 목록 CLI 앱 구축:
- 할 일 추가 기능
- 전체 할 일 목록 조회
- 일단 DB 붙이지 말고 껐다 켜면 다 날아가는 메모리(배열) 방식으로 띄워봐
```

**2단계: 살짝 기능 확장**

```
좋아, 여기까지 작동 잘하네. 이번엔 이 기능들을 추가로 덧붙여(Edit) 줘:
- 완료한 항목 체크 V 마킹 기능
- 항목 삭제 기능
```

**3단계: 견고한 데이터 이관**

```
이제 개발 서버 껐다 켜면 데이터 날아가는 거 짜증 나니까, 배열 대신 JSON 파일로 물리적으로 저장되게 데이터 로직을 바꿔 놔.
```

이렇게 작게 쪼개어 단계별(Step-by-step)로 지시해야, 어느 단계에서 버그가 터졌는지 명확히 파악하고 되돌리기가 100배 수월해집니다.

## 레슨 5: 실전에서 바로 써먹는 코딩 프롬프트 템플릿 5종

### 패턴 1: The Scaffold (초기 뼈대 구축형)

새 프로젝트나 기능을 맨땅에서 시작할 때:

```
[React TypeScript] 환경의 프로젝트를 다음 구조로 세팅해 줘:
- src/components (리액트 UI 부품들)
- src/hooks (커스텀 훅)
- src/types (TS 타입 선언 파일들)

조건: 빌드 도구는 Vite 사용, ESLint+Prettier 설정 파일 세팅, Tailwind CSS 적용
```

### 패턴 2: The Feature (기능 추가형)

이미 있는 앱에 새 모듈을 런칭할 때:

```
[유저 회원가입/로그인 인증] 기능을 내 앱에 추가해 줘:
- 이메일/비밀번호로 회원가입 및 로그인 지원할 것
- 성공 시 JWT 토큰 발생킬 것
- 비밀번호 암호화는 bcrypt 패키지 쓸 것
- 유저 데이터 로직은 데이터베이스 연동 처리
```

### 패턴 3: The Fix (버그 진압형)

에러가 나서 시스템이 박살 났을 때:

```
상황: "Cannot POST /api/login" 이라는 에러가 터짐
발생 조건: 내가 로그인 폼 버튼을 눌러 제출을 시도했을 때 터짐
정상 구동 기대치: 원래라면 상태코드 200과 함께 토큰이 들어와야 정상임

이 경로가 왜 안 먹히는지 전체 코드를 조사(Investigate)하고 수리(Fix)해 줘.
```

### 패턴 4: The Refactor (코드 청소형)

스파게티 코드를 깔끔히 청소할 때:

```
[유저 입력값 검증하는 함수 코드들]을 다음과 같은 목적으로 리팩토링 해:
- 쓸데없이 중복된 겹치는 코드들 싹 제거
- 타 프로젝트에서도 쓸 수 있게 완전 구조 분리 재사용성 확보
- 에러 반환 메시지를 예쁘고 통일되게 정리

조건: 현재 잘 돌아가고 있는 기존 검증 룰(기능)이 하나라도 훼손되거나 바뀌면 안 됨!
```

### 패턴 5: The Test (방어막 작성형)

기능은 완벽한데 무너질까 무서울 때:

```
작성된 [calculateTotal] 함수 모듈에 대한 무한 테스트 코드를 제너레이트 해:
- 양수들 합산이 정확한지 확인
- 음수값 들어갔을 때의 동작
- 아무것도 없는 빈 배열 넣었을 때 0을 잘 뱉는지
- 숫자가 아닌 기괴한 문자가 들어갔을 때 에러(Throw) 튕겨내는지

조건: Jest 프레임워크 사용, 테스트 커버리지 100% 달성 목표로 매우 깐깐하게.
```

## 레슨 6: 거꾸로 Claude Code에게 질문과 결정을 유도하기

이건 최고의 숨겨진 꿀팁입니다. 내가 어떻게 지시할지 막막할 때 AI 보고 먼저 나를 인터뷰 하라고 시켜버리세요.

```
내 앱에 사용자 개인용 대시보드(Dashboard) 화면을 크게 구축하려고 해.
네가 이걸 남김없이 완벽하게 코딩하기 위해 혹시 나한테 미리 물어봐서 확인해야 할 조건이나 알아야 할 사항들이 있다면 다 질문해 봐.
```

그러면 Claude Code가 똑똑하게 역질문 리스트를 뽑아옵니다:

- "대시보드에 구체적으로 어떤 통계 숫자들을 뿌려드릴까요?"
- "모바일 화면 반응형 크기도 챙겨드릴까요?"
- "로딩 중에 스피너나 스켈레톤 UI 같은 것 띄울까요?"

이렇게 주고받으면 내가 미처 몰랐던 치명적인 구멍들을 막고 시작할 수 있습니다.

## 자주 하는 프롬프팅 최악의 실수들

**실수 1: 단답형 지시** -- "여기 에러 고쳐". 도대체 수만 줄 파일 중 어딜 말하는 겁니까.
**실수 2: 한 번에 수십 개 지시** -- "로그인 만들고 메일 발송 붙이고 결제 모듈 통과시키고...". 중간에 꼬이면 대참사 납니다. 쪼개서 시키세요.
**실수 3: 맥락 없음** -- 내 프로젝트 환경, 폴더 구조, 프레임워크 룰도 안 알려주면서 냅다 "무조건 만들어 와" 하는 태도. 기존 코드를 모방하라 시키세요.
**실수 4: 하지 말아야 할 짓을 통제 안 함** -- "이거저거 개선해" 하면 멀쩡하던 코드 변수명까지 다 자기 멋대로 지문개조 해버립니다. "~~건 제외하고, ~~기능은 그대로 냅두고" 라고 방어막(Negative Instructions)을 꼭 치세요.
