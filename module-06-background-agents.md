# 모듈 6: 백그라운드 에이전트 다루기

**목표:** 복잡하고 여러 단계가 필요한 작업을 특화된 에이전트(Agent)에게 위임하는 비법 배우기

**예상 소요 시간:** 35-45분

---

## 무엇을 배우게 되나요?

이 모듈에서는 **에이전트(Agents)**를 다룹니다. 에이전트는 하나하나 지시하기엔 너무 벅차거나 귀찮은 거대한 복합 미션을 스스로 알아서(Autonomous) 처리하는 AI 특수 요원입니다. 언제 이 요원을 호출해야 하는지, Explore(탐험) 에이전트와 Plan(기획) 에이전트의 사용법, 그리고 시간이 없을 때 여러 요원을 동시에 달리는 병렬(Parallel) 처리 기법을 배웁니다.

---

## 레슨 1: 에이전트(Agents)란 무엇인가요?

### 에이전트 이해하기

에이전트는 독립적으로 복잡한 임무를 수행하는 특수 AI 비서입니다. 여러분이 목표만 던져주면, 스스로 어떻게 할지 스텝을 짜고, 그에 맞춰 일한 뒤, 다 치우고 나서 최종 보고만 올립니다.

기본 Claude Code와 모드가 어떻게 다른지 볼까요:

**기본 Claude Code 모드:**

- 한 걸음씩 단계별로 지시합니다.
- 말을 주고받는 대화형 핑퐁입니다.
- 매 순간의 결정권은 당신에게 있습니다.

**에이전트 모드:**

- 100% 자율적으로 움직입니다.
- 스스로 판단하고 파일을 뒤지며 결정합니다.
- 여러 단계의 거대 임무를 사람 개입 없이 수행합니다.
- 일이 다 끝난 뒤에만 나타나 보고합니다.

요약: 기본 모드는 '대화'이고, 에이전트 모드는 '외주(위임)' 입니다.

---

### 언제 에이전트를 써야 하나요?

낯선 코드베이스 전체를 스캔하거나, 새 거대 기능의 설계도를 짜거나, 수십 개의 파일을 넘나들며 리서치를 해야 하는 등, 당신이 "이거 하고... 그다음 저거 열고... 그다음 저거 찾아서..." 라고 계속 말하기 귀찮은 수준의 일일 때 씁니다.

반대로, 단순한 파일 한 줄 수정, 단일 액션, 혹은 클로드가 어떤 도구를 꺼내 쓰는지 눈으로 하나하나 확인하며 배우고 싶을 때는 기본 모드를 쓰세요.

---

## 레슨 2: Explore (탐험) 에이전트

### Explore 에이전트란?

Explore 에이전트는 거대한 코드베이스를 누비며 지형지물을 파악하는 정찰병입니다. 프로젝트가 어떻게 굴러가는지 파악하거나, 특정 로직이 어디 숨어있는지 찾아내고, 전체 아키텍처의 분위기를 읽어내는 데 특화되어 있습니다. 여러분이 파일을 일일이 열어보지 않아도, 이 녀석이 코드를 읽고 맥락을 추적해 한 편의 완벽한 설명서를 들고옵니다.

---

### Explore 에이전트 사용법

**기본 사용법:**

```
Explore 에이전트를 동원해서 이 코드베이스에서 인증(Authentication) 시스템이 어떻게 굴러가는지 다 파악해 와.
```

이렇게 시키면 에이전트는 인증 관련 파일들을 사냥개처럼 찾아내고, 내부 코드를 읽고, 보안의 흐름을 쫓아간 뒤, 당신에게 완벽한 보고서 브리핑을 바칩니다. 손 하나 까딱 안 해도 됩니다.

---

### Explore 에이전트 실전 예시

**예시 1: 거대 기능 파악하기**

```
Explore 에이전트로 결제(Payment) 처리 프로세스 좀 해부해 봐.
다음을 반드시 알아내서 보고할 것:
- 현재 어떤 결제 대행사(PG) 모듈이 붙어있는지
- 결제 기록 데이터는 어디에 어떻게 저장하는지
- 결제 쪽에 적용된 보안 로직들
- 결제창 띄울 때부터 완료 떨어질 때까지의 전체 런타임 흐름
```

**예시 2: 숨어있는 API 싹 다 캐내기**

```
코드베이스 전체를 구석구석 정찰해서 모든 API 엔드포인트(주소) 목록 장부를 만들어 와:
- HTTP 메서드 (GET/POST 등)
- 라우트 절대 경로
- 해당 API가 무슨 역할을 하는지
- 필수 첨부 파라미터(Req body)
```

**예시 3: 프로젝트 큰 그림(아키텍처) 그리기**

```
Explore 에이전트야, 이 코드베이스 스캔하고 아래 철학들을 리포트 해 와:
- 전체적인 아키텍처 패턴 (MVC인지 Layered 등인지)
- 각 부품(컴포넌트)끼리 어떻게 메시지를 주고받는지
- 데이터베이스 뼈대 구조
- 끌어다 쓴 외부 패키지(Dependencies) 목록
```

---

### 쪼는 정도(Thoroughness) 조절하기

대충 겉핥기로 볼지, 뼛속까지 파고들지 지정할 수 있습니다.

**아주 빠른 겉핥기 정찰:**

```
Explore 에이전트를 quick(빠른) 모드로 돌려서, 이 프로젝트 대충 무슨 앱인지 파악만 해줘
```

**보통 수준 정찰:**

```
Explore 에이전트를 medium(보통) 수준으로 돌려서, 로그인 시스템 파악해
```

**잡티 하나 안 놓치는 정밀 정찰:**

```
Explore 에이전트를 매우 thorough(철저한) 모드로 돌려서, API 전체 문서 개벽을 다 뜯어내서 만들어 와
```

## 레슨 3: Plan (기획/설계) 에이전트

### Plan 에이전트란?

코드 한 줄 치기 전에 "구현 전략(상세 설계도)"부터 그려오는 두뇌 역할입니다. 새로운 빅 피쳐 기능을 기획하거나, 아키텍처를 뒤엎거나, 복잡한 수정 작업을 할 때, 먼저 시뮬레이션을 돌려 지뢰(버그)를 밟지 않게 해줍니다.

진짜 명심하세요 -- 대공사 전에 Plan 에이전트와 5분만 떠들면, 길 막혀서 빙빙 돌 5시간을 아껴줍니다.

---

### Plan 에이전트 사용법

**기본 사용법:**

```
Plan 에이전트를 써서 '유저 권한/등급제(Roles and Permissions)' 기능을 추가하기 위한 상세 구현 설계도(Plan)를 작성해.
```

---

### Plan 에이전트 실전 예시

**예시 1: 새 기능 설계하기**

```
유저가 메시지를 받으면 이메일로도 알람이 쏘아지는 기능을 만들고 싶어.

Plan 에이전트를 써서 아래 항목이 모두 포함된 구현 설계도를 기획해:
- 데이터베이스에서 바꿔야/추가할 테이블
- 내가 새로 작성해야 할 코드(파일) 목록
- 건드려야 할 기존 파일들
- 테스트는 어떻게 할지 전략
- 작업하다 터질 수 있는 치명적 변수나 함정들
```

에이전트는 수 분 내에 순서가 매겨진 매우 디테일하고 무시무시한 기획안을 뱉어냅니다 (DB 업데이트 → 이메일 서비스 연동 로직 작성 → 기존 메시지 로우 훅에 연결 등).

**예시 2: 리팩토링 설계하기**

```
Plan 에이전트를 띄워서, 기존 예전 Callback 스타일로 떡칠된 DB 쿼리문들을 전부 최신 async/await 형태로 갈아엎는 전략 기획서를 써 와.
```

**예시 3: 성능 최적화(튜닝) 설계하기**

```
데이터가 많아지니까 대시보드 화면 로딩이 너무 느려터졌어.
Plan 에이전트야, 대시보드 속도를 최적화할 전략 기획을 세워 줘.
```

## 레슨 4: General-Purpose (다목적) 에이전트

### 다목적 에이전트란?

"탐색"이나 "기획"이라는 한 단어로 정의할 수 없는 짬뽕 임무를 맡습니다. 리서치(검색)와 구현 코딩이 짬뽕되어 있거나, 무자비한 버그 사냥, 코드베이스 전체 싹쓸이 일괄 업데이트 등을 알아서 수행합니다.

### 실전 예시

**예시 1: 리서치 후 셀프 구현**

```
에이전트 하나 파견해서 이거 처리해:
1. API 속도 제한(Rate limiting) 기술의 최신 트렌드를 구글링(리서치)해 와
2. 그 트렌드를 바탕으로 우리 API 엔드포인트들에 속도 제한 코드를 스스로 붙여
3. 방어 로직이 잘 도는지 테스트 코드도 짜놓고
4. 어떻게 작동하는지 문서(Docs)까지 다 남겨놔
```

**예시 2: 잠복 버그 헌터**

```
에이전트 하나 돌려서:
- 우리 인증(Auth) 시스템에 날 뚫고 들어올 수 있는 보안 취약점 싹 다 추적해
- 구멍 찾으면 네가 알아서 코드 다 수정해서 막아 놔
- 다시 안 뚫리게 방어막(테스트) 코드 치고
- 뭘 수정했는지 보고서 올려
```

## 레슨 5: 에이전트 여럿 동시 다발로 돌리기 (Parallel)

### 병렬 처리(Parallel)가 왜 쩔어주나요?

서로 연관성 없는 3개의 거대한 질문을 굳이 1개씩 순서대로 물어볼 필요가 없습니다. 여러 명의 AI 요원을 동시다발로 백그라운드에 풀어서 알아서 각자 일하게 던져두면 시간을 비약적으로 아낍니다.

---

### 병렬 처리 지시법

**이렇게 시킵니다:**

```
아래 임무들 각각 별도의 에이전트 파견해서 병렬(Parallel)로 동시에 굴려:

Agent 1: 인증 시스템 아키텍처 죄다 탐색해서 보고하기
Agent 2: 결제 처리 시스템 죄다 탐색해서 보고하기
Agent 3: 현재 존재하는 모든 API 주소 리스트 뽑기
```

---

### 유용한 병렬 처리(Parallel) 실전 팁

**기획안 3개 한 번에 짜기:**

```
병렬로 돌려:
1. 유저 권한 기능 추가하는 기획안 짜는 에이전트
2. 이메일 알림 추가하는 기획안 짜는 에이전트
3. 접속 로그(Audit log) 쌓는 기획안 짜는 에이전트
```

**외주 리서치 여러 개 동시 돌리기:**

```
아래 내용 병렬로 리서치 해 와:
1. REST API 최신 디자인 최고 모범 사례
2. 데이터베이스 인덱싱(Indexing) 최적화 방법론
3. 서버 Caching 전략 모범 사례

다 끝나면 세 개 합쳐서 나한테 요약 브리핑 해.
```

## 레슨 6: 에이전트가 물고 온 결과물 써먹기

### 에이전트 보고서 이해하기

에이전트는 끝날 때 무지막지하게 화려한 보고서(설명, 파일 위치, 코드 조각, 개선점, 경고)를 들고옵니다. 이건 그냥 읽고 넘기는 게 아니라 써먹으라고(Actionable) 들고 오는 겁니다.

### 기획안(Plan)을 액션으로 옮기기

에이전트가 거대한 설계도를 짜왔다면 응답하세요.

**허락 후 런칭:**

```
기획안 죽이네. 그대로 1단계(데이터베이스 마이그레이션)부터 코드 짜서 구현 실시해.
```

**태클 걸기 (수정 지시):**

```
기획안은 좋은데 나 백그라운드 잡(Job) 돌리는 건 싫어. 그냥 동기(Sync) 처리로 이메일 쏘는 방식으로 기획안 다시 뜯어 고쳐봐.
```

**이해가 안 갈 때 질문:**

```
기획안 3단계 부분이 무슨 개소린지 좀 더 쉽게 상세히 풀어 설명해 줄래?
```

## 레슨 7: 에이전트 프로들의 사용 수칙(Best Practices)

### 진짜 제발 구체적으로 지시하세요

에이전트가 헛발질하는 건 100% 프롬프트가 모호해서입니다.

❌ **최악:**

```
코드베이스 좀 탐험해 봐
```

✅ **최고:**

```
코드베이스 전역을 탐험해서 모든 데이터베이스 쿼리문들을 솎아 낸 다음에, 특별히 속도가 느려 터질 수 있거나 N+1 쿼리 에러 낼 만한 위험 지역을 집중적으로 파악해.
```

### 목표를 명확(Clear)하게

❌ **최악:**

```
시스템 좀 개선할 기획 좀 짜봐
```

✅ **최고:**

```
데이터베이스 부하를 줄이기 위해 API에 캐싱(Caching)을 덮으려고 해. 앱에서 제일 빡세게 리퀘스트 들어오는 순위 5개 API 엔드포인트만 노려서 캐싱을 먹이는 기획안을 설계해.
```

### 당신의 한계(Constraints)를 알려라

현실 프로젝트는 제약 조건 투성이입니다. 미리 말해주세요:

```
실시간 알람 시스템을 짤 기획안을 세워보는데, 이 제약조건 명심해:
- 사내 방화벽 땜에 WebSockets 절대 쓰면 안 됨
- 지금 쓰는 DB (SQLite) 환경 내에서만 해결 돌릴 것
- 나중에 또 세팅하기 빡세니까 배포하기 아주 심플해야 함
```

## 모듈 6 요점 체크리스트

모듈 7로 넘어가기 전 점검:

- [ ] 에이전트가 단순 대화형 모드와 어떻게 다른지 안다.
- [ ] 낯선 코드나 시스템 파악을 위해 'Explore 에이전트'를 군견처럼 풀 수 있다.
- [ ] 구현 전 엉성한 실수를 막기 위해 'Plan 에이전트'로 완벽한 설계도를 먼저 짤 수 있다.
- [ ] 시간 단축을 위해 서로 상관없는 일들을 '병렬(Parallel) 에이전트'로 흩뿌릴 수 있다.
- [ ] 에이전트에게 뭉뚱그려 말하지 않고 명확한 제약 조건과 목표를 지시할 줄 안다.
